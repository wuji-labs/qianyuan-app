# PI and ACP Backend Feature Matrix

This is a low-level implementation matrix for PI and the ACP-oriented backends that are candidates for catalogization.

Legend:

- `native`: provider keeps a provider-owned runtime/control plane
- `bespoke ACP`: provider uses ACP, but still owns backend/runtime glue
- `catalog+hook`: provider can ride the generic ACP catalog path plus a thin hook layer
- `missing`: no shipped surface today

Common cross-cutting files:

- Catalog/runtime spine: `apps/cli/src/agent/acp/runtime/createCatalogProviderAcpRuntime.ts`, `apps/cli/src/agent/acp/createCatalogAcpBackend.ts`
- Built-in catalog-defined ACP path: `apps/cli/src/agent/acp/catalog/createCatalogDefinedAcpEntry.ts`, `apps/cli/src/agent/acp/catalog/createCatalogDefinedAcpBackend.ts`, `apps/cli/src/agent/acp/catalog/runCatalogDefinedAcpAgent.ts`
- Built-in ACP metadata: `packages/agents/src/acp.ts`, `packages/agents/src/manifest.ts`, `packages/agents/src/auth.ts`, `packages/agents/src/providers/providerCliRuntime.ts`
- Backend selection in UI: `apps/ui/sources/agents/backendCatalog/getResolvedBackendCatalogEntries.ts`, `apps/ui/sources/components/sessions/new/hooks/useNewSessionScreenModel.tsx`, `apps/ui/sources/__tests__/routes/(app)/session/[id]/runs/new.test.tsx`
- Execution-run registry: `apps/cli/src/agent/executionRuns/registry/executionRunBackendRegistry.ts`
- ACP catalog QA: `packages/tests/suites/ui-e2e/settings.acpCatalog.spec.ts`

## Migration notes by bucket

### Keep native / bespoke

- PI: keep provider-owned until generic hooks can express RPC transport, PI thinking, explicit tool menus, and deterministic `--session` restarts.
- Gemini: keep provider-owned because model/auth/MCP/runtime behavior is substantially richer than the thin ACP wrappers.

### Best catalog-plus-hook candidates

- Qwen
- Kilo
- Copilot
- Auggie
- Kimi

Shared migration target for that group:

- Use the generic catalog runtime path from `apps/cli/src/agent/acp/runtime/createCatalogProviderAcpRuntime.ts`.
- Replace provider-owned `index.ts` / `run*.ts` glue with a catalog entry plus small hooks for spawn args, transport, auth-launch, and permission mapping.
- Keep provider-specific session-id metadata helpers until metadata publication is unified.
- Preserve execution-run registration in `apps/cli/src/agent/executionRuns/registry/executionRunBackendRegistry.ts` while moving factories behind catalog descriptors.

### Already effectively catalog-defined

- Kiro: treat it as the reference implementation for built-in catalog ACP. The gap is not architecture, it is missing surfaces: execution-run registration, stronger tests, and a deliberate decision on whether direct-session UX should exist.

## Biggest current gaps

- Copilot and Kiro have no provider harness spec file under `apps/cli/src/backends/*/e2e/providerSpec.json`.
- Kiro has no execution-run registration.
- Most catalogizable ACP providers still ship `createUnknownCliAuthSpec(...)` for auth-status detection, even though native login-launch flows are already implemented for Qwen, Kilo, Kimi, and Auggie.
- None of PI, Qwen, Kilo, Copilot, Auggie, Kimi, or Gemini currently ship a direct-session browser/import surface comparable to Codex / Claude / OpenCode.
