# PI Feature Matrix

This document captures the current low-level PI implementation across CLI, UI, shared agent metadata, and provider test surfaces. It is an internal architecture inventory for the unified-architecture migration, not end-user documentation.

## Status legend

- `supported`: primary shipped path
- `partial`: shipped, but only at one layer or with important product gaps
- `unsupported`: intentionally not exposed today

## Runtime surfaces

| Feature | Status | Exact source files | Current behavior | Unified-architecture migration notes |
| --- | --- | --- | --- | --- |
| Custom RPC runtime | `supported` | `apps/cli/src/backends/pi/index.ts`, `apps/cli/src/backends/pi/runPi.ts`, `apps/cli/src/backends/pi/acp/backend.ts`, `apps/cli/src/backends/pi/acp/runtime.ts`, `apps/cli/src/backends/pi/rpc/PiRpcBackend.ts` | PI ships as a bespoke ACP-shaped provider, but the transport is PI-owned RPC rather than generic ACP JSON-RPC. `runPi()` still uses `runStandardAcpProvider(...)`, while backend creation ultimately instantiates `PiRpcBackend`. | Keep PI provider-owned until the generic runtime can express a non-ACP transport, provider-owned restart semantics, and PI-specific runtime state publication without pretending the transport is ordinary catalog ACP. |

## Cross-cutting migration notes

1. PI is not a plain catalog ACP backend. The user-session shell looks shared, but transport, restart behavior, and runtime state publication are PI-owned in `apps/cli/src/backends/pi/rpc/PiRpcBackend.ts`.
2. PI has a real low-level resume primitive today, but shared product policy still marks PI vendor resume unsupported. That is the biggest capability mismatch in the current stack.
3. PI permissions are tool-whitelist driven, not MCP-prompt driven. The unified architecture needs an explicit provider policy surface for tool menus, not just generic permission modes.
4. PI auth is split across three layers: env-only auth detection, connected-service `auth.json` materialization, and backend idle-boundary restart when `auth.json` changes. Those should become one declared provider-auth contract.
5. PI controls are currently fragmented: runtime model/command introspection comes from RPC, thinking is a provider-specific UI startup option, and session modes are intentionally absent. A unified controls model should represent those as separate declared capabilities instead of ACP-shaped metadata overload.
