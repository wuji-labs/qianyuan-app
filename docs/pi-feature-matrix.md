# PI Feature Matrix

This document captures the current low-level PI implementation across CLI, UI, shared agent metadata, and provider test surfaces. It is an internal architecture inventory for the unified-architecture migration, not end-user documentation.

## Audit evidence snapshot

- Happier source audited in `remote-dev` on 2026-05-20.
- Installed PI: `pi` on PATH, version `0.73.1`.
- Upstream PI source clone audited at `/tmp/pi-audit-clone`.
- Local PI RPC command/skill probe used `pi --mode rpc --no-extensions` with a temporary `PI_CODING_AGENT_DIR`.
- Local PI RPC fork/clone probe used a seeded JSONL session in a temporary PI directory.

## Status legend

- `supported`: primary shipped path
- `partial`: at least one required layer exists, but product support is incomplete or mismatched
- `unsupported`: intentionally not exposed today
- `unconfirmed`: source hints exist, but no stable RPC/CLI contract or local probe proves product-ready support

## Recommended feature-support backlog

1. **Command picker / skills / prompt templates** — support via PI RPC `get_commands`. First fix/test the Happier adapter shape so `name` reaches `normalizeAvailableCommands(...)`; then decide whether to preserve PI `source`/`sourceInfo` in shared slash-command metadata.
2. **Fork / clone** — support is feasible via PI RPC, but should be a separate TDD implementation after lifecycle correctness. Add RED tests for `get_fork_messages`, `clone`, and `fork` mapping before changing `packages/agents/src/manifest.ts`.
3. **Rollback** — keep unsupported. Internal PI tree navigation is not enough; require a top-level stable RPC/CLI command and real probe evidence.
4. **Continuation command semantics** — real PI RPC probing showed standalone `follow_up` queues silently after idle; compaction auto-continuation should use `prompt` with `streamingBehavior: 'followUp'`, which did restart work after `compaction_end`.
5. **Thinking controls** — refine model-aware PI thinking options only if RPC model metadata is sufficient; avoid SDK-only assumptions.

## Cross-cutting migration notes

1. PI is not a plain catalog ACP backend. The user-session shell looks shared, but transport, restart behavior, and runtime state publication are PI-owned in `apps/cli/src/backends/pi/rpc/PiRpcBackend.ts`.
2. PI has a real low-level resume primitive and shared product policy now marks PI vendor resume supported when metadata includes a vendor id.
3. PI permissions are tool-whitelist driven, not MCP-prompt driven. The unified architecture needs an explicit provider policy surface for tool menus, not just generic permission modes.
4. PI auth is split across three layers: env-only auth detection, connected-service `auth.json` materialization, and backend idle-boundary restart when `auth.json` changes. Those should become one declared provider-auth contract.
5. PI controls are fragmented: runtime model/command introspection comes from RPC, thinking is a provider-specific option, and session modes are intentionally absent. A unified controls model should represent those as separate declared capabilities instead of ACP-shaped metadata overload.
6. PI command, skill, prompt, and fork support should be RPC-backed. Do not switch Happier to direct PI SDK embedding for these capability gaps unless a separate architecture decision explicitly accepts SDK coupling.
